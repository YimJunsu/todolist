1. OT(Operational Transformation)란 무엇인가?

Operational Transformation(이하 OT)은 여러 사용자가 동시에 같은 문서를 편집할 때 발생하는 충돌을 "거부(reject)" 없이 "변환(transform)"으로 해결하는 알고리즘이다. 구글 독스(Google Docs), 이전의 Google Wave, Firepad 등 협업 에디터의 핵심 원리로 사용된다.

OT의 핵심 아이디어는 단순하다. 두 사용자가 동시에 같은 문서에 편집을 가했을 때, 한쪽의 작업을 상대방의 작업이 이미 적용된 상태에 맞게 위치를 보정(transform)해서 두 작업을 모두 반영하는 것이다.

핵심 개념 정리

  • Operation(작업): 문서에 가하는 최소 단위 변경 — 삽입(insert) 또는 삭제(delete)
  • revision: 서버가 작업을 누적 적용한 횟수 (버전 번호와 유사하나, 충돌 해결의 기준으로 사용)
  • transform(op1, op2): op2가 먼저 적용된 문서에서 op1을 올바르게 적용하기 위해 op1의 위치를 보정하는 함수

1-1. 직관적인 예시

아래 상황을 가정한다.

  • 문서 현재 내용: "hello"
  • 사용자 A: 3번 위치에 'X' 삽입 → "helXlo"
  • 사용자 B (동시에): 3번 위치에 'Y' 삽입 → "helYlo" 의도

두 작업이 동시에 서버에 도달했을 때, A의 작업이 먼저 처리된다면 문서는 "helXlo"가 된다. 이 상태에서 B의 작업(3번 위치에 'Y' 삽입)을 그대로 적용하면 결과는 "helYXlo"가 되어 버린다. B의 의도는 X 뒤가 아닌 원래 3번 위치였으므로, B의 작업을 A의 영향을 반영해 4번 위치로 보정해야 한다. 

A: insert(3, 'X')  →  먼저 적용됨 → "helXlo"
B: insert(3, 'Y')  →  transform(B, A) → insert(4, 'Y') → "helXYlo"

최종 결과: "helXYlo" — 두 입력 모두 반영됨

2. 기존 버전 방식의 문제점과 한계

OT 도입 이전, 이 프로젝트는 버전 번호(revision) 기반 충돌 방지 방식을 사용하였다. 구조는 단순하다. 클라이언트가 작업을 보낼 때 자신이 알고 있는 서버의 버전 번호를 함께 보내고, 서버가 해당 버전이 현재 서버 버전과 다르면 "conflict(충돌)"로 판단해 해당 작업을 거부하는 방식이다.

2-1. 기존 충돌 처리 흐름

클라이언트 A: "3번 위치에 'X' 삽입" (revision=3 기반으로 전송)
클라이언트 B: "3번 위치에 'Y' 삽입" (revision=3 기반으로 동시에 전송)

서버: A의 작업을 먼저 받아 처리 → serverRevision = 4
      B의 작업이 도착했을 때: B.revision(3) ≠ serverRevision(4)
      → "conflict" 메시지 전송
      → B의 내용이 사라짐

2-2. 기존 방식의 핵심 한계

  • 동시 편집 시 데이터 유실 필연적: 두 사용자가 동시에 입력하면 네트워크 지연 순서에 따라 한 명의 작업이 항상 거부된다. 빠른 타이핑 상황에서는 연속적인 유실이 발생한다.
  • 사용자 경험 저하: 충돌이 발생할 때마다 클라이언트에 경고 배너를 표시하거나, 사용자가 직접 새로고침해서 최신 내용을 다시 불러와야 한다.
  • 재전송 로직 부재: 거부된 작업을 어떻게 재처리할지 전략이 없다. 단순히 "충돌이 발생했다"는 사실만 알릴 뿐, 자동 복구 메커니즘이 없다.
  • 버전 번호만으로는 "어디서 충돌이 발생했는가"를 알 수 없다: 버전이 달라도 실제로는 겹치지 않는 위치를 편집했을 수 있다. 버전 번호 방식은 이를 구분하지 못하고 무조건 거부한다.

2-3. 구체적 문제 시나리오

시나리오: 두 사용자가 다른 위치를 편집해도 충돌 발생

문서: "Hello World" (revision=5)
A: 0번 위치에 "Hi " 삽입 (완전히 앞 부분)
B: 11번 위치에 "!" 삽입 (완전히 뒷 부분, 동시에 전송)

→ 두 작업은 서로 다른 위치를 건드리므로 실제로는 충돌이 없다.
→ 그럼에도 버전 번호 방식에서는 한 명이 무조건 거부당한다.

3. OT 방식 전환 과정

3-1. 설계 방향 결정

OT를 구현하는 방법은 다양하다. ShareDB, Quill Delta, Automerge(CRDT 계열) 등 라이브러리도 있으나, 이 프로젝트에서는 의존성을 최소화하고 알고리즘을 직접 이해하는 것을 목표로 순수 TypeScript로 OT 유틸리티를 직접 구현하였다.

편집 작업은 두 종류로 단순화한다.

  • InsertOp: 특정 위치에 문자열을 삽입한다.
  • DeleteOp: 특정 위치에서 특정 길이만큼 문자를 삭제한다.

3-2. 핵심 변환 케이스 도출

transformOp(op1, op2)는 op2가 먼저 적용된 상태에서 op1을 올바르게 적용하기 위해 op1의 위치를 보정하는 함수이다. op1과 op2의 타입 조합에 따라 4가지 케이스를 처리한다.

케이스 상황 보정 규칙
Insert vs Insert op2가 op1 앞에 삽입 op1 위치를 op2.text.length만큼 오른쪽으로 이동
Insert vs Delete op2가 op1 앞 텍스트를 삭제 op1 위치를 삭제 길이만큼 왼쪽으로 이동. 삭제 범위가 op1 위치를 포함하면 삭제 시작점으로 이동
Delete vs Insert op2가 op1 삭제 범위 안에 삽입 삭제 길이를 삽입된 텍스트 길이만큼 늘린다 (삽입된 텍스트도 삭제 대상에 포함)
Delete vs Delete 두 삭제 범위가 겹침 이미 삭제된 부분을 제외하고 남은 범위만 삭제. 완전히 포함된 경우 length=0

3-3. 클라이언트 상태 머신 설계

클라이언트는 3개의 상태 변수로 OT 상태 머신을 구성한다.

  • serverRevision: 마지막으로 서버가 승인(ack)한 revision 번호
  • inflightOps: 서버에 전송했으나 아직 ack를 받지 못한 작업 목록
  • bufferOps: 로컬에서 생성됐으나 아직 서버에 보내지 않은 작업 목록

이 구조를 통해 네트워크 지연 중에도 사용자의 입력이 화면에 즉시 반영되고, 서버 응답(ack 또는 remote op)을 받은 후 순서를 정확히 조정할 수 있다.

3-4. textarea diff → TextOp 변환

React의 textarea onChange 이벤트는 변경 전/후 전체 텍스트만 제공한다. 이를 InsertOp 또는 DeleteOp로 변환하기 위해 diffToOps 함수를 구현하였다. 

// client/src/ot.ts — diffToOps 알고리즘
export function diffToOps(oldContent: string, newContent: string): TextOp[] {
  // 1. 앞에서부터 공통 부분(prefix) 계산
  let start = 0;
  while (start < oldContent.length && start < newContent.length
         && oldContent[start] === newContent[start]) {
    start++;
  }

  // 2. 뒤에서부터 공통 부분(suffix) 계산 (start 이후 범위에서만)
  let oldEnd = oldContent.length;
  let newEnd = newContent.length;
  while (oldEnd > start && newEnd > start
         && oldContent[oldEnd - 1] === newContent[newEnd - 1]) {
    oldEnd--;
    newEnd--;
  }

  const ops: TextOp[] = [];

  // 3. 구버전에만 있는 부분 → 삭제
  if (oldEnd > start) {
    ops.push({ type: 'delete', position: start, length: oldEnd - start });
  }

  // 4. 새버전에만 있는 부분 → 삽입 (삭제 후 삽입이므로 position은 start 그대로)
  if (newEnd > start) {
    ops.push({ type: 'insert', position: start, text: newContent.slice(start, newEnd) });
  }

  return ops;
}

공통 앞부분(prefix)과 공통 뒷부분(suffix)을 제외하면 중간에 변화한 구간만 남는다. 그 구간에서 구버전 텍스트가 있으면 delete, 신버전 텍스트가 있으면 insert를 생성한다.

4. OT 작동 흐름 (현재 소스 기준)

4-1. 전체 흐름 순서도

아래 순서도는 클라이언트 A가 텍스트를 입력하고 서버를 통해 클라이언트 B에 전파되는 전체 흐름을 나타낸다.

4-2. 데이터 타입 정의

서버(server/src/ot.ts)와 클라이언트(client/src/ot.ts) 모두 동일한 타입을 정의한다.

// 삽입 작업: position 위치에 text를 삽입한다
type InsertOp = {
  type: 'insert';
  position: number;  // 삽입할 위치 (0-indexed)
  text: string;      // 삽입할 텍스트
};

// 삭제 작업: position 위치에서 length 글자를 삭제한다
type DeleteOp = {
  type: 'delete';
  position: number;  // 삭제 시작 위치 (0-indexed)
  length: number;    // 삭제할 글자 수
};

type TextOp = InsertOp | DeleteOp;

// 클라이언트가 서버로 보내는 작업 메시지
type ClientOp = TextOp & {
  revision: number;  // 클라이언트가 기반하는 서버 revision 번호
  opId: string;      // 작업 고유 ID — 서버 ack 수신 시 매칭에 사용
};

4-3. applyOp: 작업을 문서에 적용

applyOp는 문자열에 TextOp를 적용하고 결과 문자열을 반환하는 순수 함수이다. 서버와 클라이언트 모두에서 동일하게 사용한다. 

// server/src/ot.ts (클라이언트와 동일 로직)
export function applyOp(content: string, op: TextOp): string {
  if (op.type === 'insert') {
    // 삽입: [앞부분] + [삽입텍스트] + [뒷부분]
    return content.slice(0, op.position) + op.text + content.slice(op.position);
  } else {
    // 삭제: [앞부분] + [삭제 이후 뒷부분]
    return content.slice(0, op.position) + content.slice(op.position + op.length);
  }
}

4-4. transformOp: 핵심 변환 함수

transformOp(op1, op2)는 op2가 먼저 적용된 상태에서 op1을 올바르게 적용할 수 있도록 op1의 위치를 보정한다. 4가지 타입 조합(insert×insert, insert×delete, delete×insert, delete×delete)을 각각 처리한다. 

// server/src/ot.ts
export function transformOp(op1: TextOp, op2: TextOp): TextOp {
  if (op1.type === 'insert' && op2.type === 'insert') {
    return transformInsertInsert(op1, op2);
  } else if (op1.type === 'insert' && op2.type === 'delete') {
    return transformInsertDelete(op1, op2);
  } else if (op1.type === 'delete' && op2.type === 'insert') {
    return transformDeleteInsert(op1, op2);
  } else {
    return transformDeleteDelete(op1 as DeleteOp, op2 as DeleteOp);
  }
}

// [삽입 vs 삽입]
// op2가 op1 앞(또는 같은 위치)에 삽입하면 op1을 오른쪽으로 밀어낸다
function transformInsertInsert(op1: InsertOp, op2: InsertOp): InsertOp {
  if (op2.position <= op1.position) {
    return { ...op1, position: op1.position + op2.text.length };
  }
  return op1;
}

// [삽입 vs 삭제]
// op2 삭제 범위가 op1 위치 앞에 있으면 op1 위치를 왼쪽으로 당긴다
function transformInsertDelete(op1: InsertOp, op2: DeleteOp): InsertOp {
  const op2End = op2.position + op2.length;
  if (op2End <= op1.position) {
    return { ...op1, position: op1.position - op2.length };
  } else if (op2.position >= op1.position) {
    return op1;
  }
  return { ...op1, position: op2.position }; // 삭제 범위가 op1 위치를 포함
}

// [삭제 vs 삽입]
// op2가 삭제 범위 안에 삽입되면 삭제 길이를 늘린다
function transformDeleteInsert(op1: DeleteOp, op2: InsertOp): DeleteOp {
  if (op2.position <= op1.position) {
    return { ...op1, position: op1.position + op2.text.length };
  } else if (op2.position >= op1.position + op1.length) {
    return op1;
  }
  return { ...op1, length: op1.length + op2.text.length };
}

// [삭제 vs 삭제]
// 두 삭제 범위가 겹치는 경우 이미 삭제된 부분을 제외하고 남은 범위만 삭제한다
function transformDeleteDelete(op1: DeleteOp, op2: DeleteOp): DeleteOp {
  const op1End = op1.position + op1.length;
  const op2End = op2.position + op2.length;

  if (op2End <= op1.position)               return { ...op1, position: op1.position - op2.length };
  if (op2.position >= op1End)               return op1;
  if (op2.position <= op1.position && op2End >= op1End) return { ...op1, position: op2.position, length: 0 };
  if (op2.position >= op1.position && op2End <= op1End) return { ...op1, length: op1.length - op2.length };
  if (op2.position < op1.position)          return { ...op1, position: op2.position, length: op1End - op2End };
  return { ...op1, length: op2.position - op1.position };
}

4-5. 서버: 동시 작업 처리 흐름

서버는 모든 클라이언트의 작업을 순서대로 처리하며, 각 작업이 도착했을 때 해당 클라이언트가 모르는 (이미 서버에서 처리된) 작업들에 대해 transformAgainstAll을 수행한다. 

// server/src/index.ts

// 서버 상태
let editorContent = '';
let serverRevision = 0;
const history: TextOp[] = [];  // 서버가 적용한 모든 작업 이력

wss.on('connection', (ws: WebSocket) => {
  // 새 접속자에게 현재 문서 상태 전송
  ws.send(JSON.stringify({ type: 'init', content: editorContent, revision: serverRevision }));

  ws.on('message', (data: Buffer) => {
    const message = JSON.parse(data.toString());

    if (message.type === 'op' && message.op) {
      const clientOp = message.op;

      // 클라이언트가 모르는 작업들 (클라이언트 revision 이후에 서버에서 처리된 작업들)
      const concurrentOps = history.slice(clientOp.revision);

      // 클라이언트 작업을 concurrent ops에 대해 순차 변환
      const transformedOp = transformAgainstAll(clientOp, concurrentOps);

      // 변환된 작업을 서버 문서에 적용
      editorContent = applyOp(editorContent, transformedOp);
      serverRevision += 1;
      history.push(transformedOp);

      // 작업 보낸 클라이언트에게 ack 전송
      ws.send(JSON.stringify({ type: 'ack', opId: clientOp.opId, revision: serverRevision }));

      // 나머지 클라이언트에게 변환된 작업 브로드캐스트
      clients.forEach((client) => {
        if (client !== ws && client.readyState === WebSocket.OPEN) {
          client.send(JSON.stringify({ type: 'op', op: transformedOp, revision: serverRevision }));
        }
      });
    }
  });
});

4-6. 클라이언트: OT 상태 머신 구현

클라이언트는 inflightOps와 bufferOps 두 큐로 OT 상태를 관리한다.

로컬 편집 시 처리 (handleChange)

// client/src/pages/EditorPage.tsx

const handleChange = (e: React.ChangeEvent<HTMLTextAreaElement>) => {
  const newContent = e.target.value;
  const oldContent = contentRef.current;
  if (newContent === oldContent) return;

  // 이전↔새 텍스트 diff로 TextOp[] 생성
  const ops = diffToOps(oldContent, newContent);
  if (ops.length === 0) return;

  // 로컬 문서를 즉시 업데이트 (사용자 반응성 확보)
  contentRef.current = newContent;
  setContent(newContent);

  // bufferOps에 추가
  bufferOpsRef.current.push(...ops);

  // inflight가 없으면 즉시 서버로 전송
  flushBuffer();
};

ack / remote op 수신 처리

// client/src/pages/EditorPage.tsx — onmessage 처리

// ── ack 수신: 내가 보낸 작업이 서버에서 승인됨 ──
if (message.type === 'ack') {
  // 해당 opId를 inflightOps에서 제거
  inflightOpsRef.current = inflightOpsRef.current.filter(op => op.opId !== message.opId);
  serverRevisionRef.current = message.revision;

  // inflight가 모두 처리됐으면 bufferOps 전송
  if (inflightOpsRef.current.length === 0) {
    flushBuffer();
  }
}

// ── remote op 수신: 다른 사용자의 작업을 반영 ──
if (message.type === 'op') {
  let remoteOp: TextOp = message.op;
  serverRevisionRef.current = message.revision;

  // 1단계: inflightOps에 대해 remoteOp를 변환 (내 inflight 작업이 먼저 적용된 상태 기준)
  for (const inflightOp of inflightOpsRef.current) {
    remoteOp = transformOp(remoteOp, inflightOp);
  }

  // 2단계: 원본 remoteOp에 대해 각 inflightOp도 변환
  let prevRemote = message.op as TextOp;
  inflightOpsRef.current = inflightOpsRef.current.map(inflightOp => {
    const newInflight = transformOp(inflightOp, prevRemote) as typeof inflightOp;
    prevRemote = transformOp(prevRemote, inflightOp);
    return newInflight;
  });

  // 3단계: bufferOps도 변환된 remoteOp에 대해 transform
  bufferOpsRef.current = bufferOpsRef.current.map(bufOp => {
    const transformed = transformOp(bufOp, remoteOp);
    remoteOp = transformOp(remoteOp, bufOp);
    return transformed;
  });

  // 4단계: 변환된 remote op를 로컬 문서에 적용
  const newContent = applyOp(contentRef.current, remoteOp);
  contentRef.current = newContent;
  setContent(newContent);
}

왜 inflightOps와 bufferOps 모두 transform해야 하는가?

서버에서 다른 사용자의 작업(remoteOp)이 이미 문서에 적용된 상태로 브로드캐스트된다. 클라이언트는 이 변경된 문서 위에서 자신의 pending 작업(inflight/buffer)을 나중에 적용해야 한다. 따라서 pending 작업들의 위치도 remoteOp의 영향을 받아 보정되어야 한다.
